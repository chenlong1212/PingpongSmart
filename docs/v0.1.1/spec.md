# v0.1.1 Spec

## 背景
v0.1.0 是一个极简的 PingpongSmart 聊天应用，包含 Vue 3 前端和 Spring Boot 后端，核心链路是「用户提问 → LLM 流式回答 → 页面展示」。该版本明确排除了持久化、会话管理、上下文记忆等功能以保持轻量。

当前状态：
- 前后端代码已就位，无数据库依赖
- 所有消息存在前端内存中，刷新页面即丢失
- 每次请求只发送当前一条消息，LLM 没有多轮对话记忆
- 后端 `docs/v0.1.0/` 目录已有上一版本的 spec/plan/task/checklist 文档

## 目标
- 引入 MySQL 持久化，刷新页面后对话记录不丢失
- 实现多轮对话上下文管理，LLM 能基于历史对话做出连贯回答

## 功能需求
- F1: Session 管理 — 为每次对话分配唯一 Session ID，存 localStorage；可新建/切换会话
- F2: 消息持久化 — 用户消息和系统回复写入 MySQL，含 session_id、role、content、created_at
- F3: 历史加载 — 按 Session ID 拉取最近 N 条消息（默认 20 条，可配置）
- F4: 多轮上下文 — 发送新消息时携带最近 N 轮对话历史（默认 10 轮，可配置）作为上下文发给 LLM
- F5: 用户标识 — 首次输入用户名，存 localStorage，用于区分不同用户

## 非功能需求
- N1: 现有单条消息发送 + SSE 流式回复功能继续正常工作
- N2: 历史消息查询使用 LIMIT，响应时间 < 500ms（本地 MySQL）
- N3: 历史加载上限和上下文轮数通过配置文件控制

## 不做的事
- 不引入用户注册/登录系统
- 不做跨设备同步，数据仅存储在本地 MySQL
- 不清理功能、归档会话等会话管理功能
- 不做消息搜索、会话列表等历史查看功能
- 不实现真正的长期记忆

## 验收标准
- AC1: 刷新页面后 Session ID 不变，对话继续
- AC2: 新建对话生成新 Session ID，旧对话可切换
- AC3: 消息发送后 MySQL 中有对应记录
- AC4: 刷新页面后最近 20 条消息正确展示
- AC5: 新消息请求携带最近 10 轮上下文，LLM 给出连贯回答
- AC6: 历史少于 10 轮时，所有历史作为上下文
- AC7: 首次进入显示用户名输入，下次自动显示
- AC8: 现有核心功能不被破坏
- AC9: 历史查询使用 LIMIT，响应 < 500ms
